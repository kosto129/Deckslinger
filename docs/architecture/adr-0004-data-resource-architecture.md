## Context

### Problem Statement
Deckslinger has 5 categories of game data that must be editor-authorable, data-driven, and hot-reloadable: cards (CardData + CardEffect), status effects (StatusEffectData), enemy behaviors (EnemyBehaviorData + EnemyAttackData), room metadata (RoomData), and run state (RunState). We need a consistent data architecture that uses Godot's Resource system effectively.

### Requirements
- All gameplay values must be data-driven (never hardcoded in scripts)
- Card data must be hot-reloadable during development
- Resources with nested sub-resources (CardData → Array[CardEffect]) must support deep copying
- Runtime-only data (HitData, draft choices) should not persist to disk

## Decision

**Use Godot Resources (.tres) for all static game data and RefCounted for runtime-only packets. Nested resources require `duplicate_deep()` (4.5+) when per-instance copies are needed.**

### Data Classification

| Category | Base Class | File Format | Persistence | Example |
|----------|-----------|-------------|-------------|---------|
| **Static Game Data** | `Resource` | `.tres` files | On disk, editor-authorable | CardData, StatusEffectData, EnemyBehaviorData |
| **Runtime Packets** | `RefCounted` | Not saved | Garbage-collected | HitData, RewardData |
| **Run State** | `Resource` | Not saved (MVP) | In-memory only (Save/Load adds serialization post-MVP) | RunState |
| **Shared Enums** | `const` / `enum` | `.gd` script | Compiled | EntityType, CardArchetype, Rarity |

### Resource Definitions

```gdscript
# Static — saved as .tres in assets/data/
class_name CardData extends Resource
class_name CardEffect extends Resource          # nested inside CardData.effects array
class_name StatusEffectData extends Resource
class_name EnemyBehaviorData extends Resource
class_name EnemyAttackData extends Resource     # nested inside EnemyBehaviorData.attacks array
class_name RoomData extends Resource            # attached to room scenes

# Runtime — never saved
class_name HitData extends RefCounted
class_name RewardData extends RefCounted
class_name FloorData extends RefCounted         # generated by Dungeon Generation
class_name WaveData extends Resource            # authored per room, part of RoomData
class_name EnemySpawnEntry extends Resource     # nested in WaveData

# Persistence
class_name RunState extends Resource            # in-memory during run, serializable post-MVP
```

### Deep Copy Rule

When runtime code needs a mutable copy of a Resource with nested sub-resources (e.g., modifying a CardEffect at runtime), use `duplicate_deep()`:

```gdscript
# CORRECT (4.5+): deep copies nested Array[CardEffect]
var card_copy: CardData = original_card.duplicate_deep()

# WRONG: shallow copy — nested CardEffects are shared references
var card_copy: CardData = original_card.duplicate()
```

For MVP, CardData resources are read-only at runtime — no duplication needed. `duplicate_deep()` becomes necessary if card upgrades modify effect values at runtime (post-MVP).

### Enum Consolidation

All cross-system enums live in a single autoload script:

```gdscript
# src/core/enums.gd — autoload as "Enums"
class_name Enums

enum EntityType { PLAYER, STANDARD_ENEMY, ELITE_ENEMY, BOSS, PROJECTILE, PROP }
enum LifecycleState { INACTIVE, SPAWNING, ACTIVE, STUNNED, DYING, DEAD }
enum CardArchetype { GUNSLINGER, DRIFTER, OUTLAW, NEUTRAL }
enum CardRarity { COMMON, UNCOMMON, RARE, LEGENDARY }
enum CardType { ATTACK, SKILL, POWER }
enum EffectType { DAMAGE, HEAL, APPLY_STATUS, MOVE_SELF, SPAWN_PROJECTILE, AOE_DAMAGE, SHIELD, DRAW_CARDS, DISCARD }
enum TargetingMode { SELF, DIRECTIONAL, AIMED, AOE_SELF, NONE }
enum EffectCategory { BUFF, DEBUFF, DOT, CONTROL }
enum StackingRule { REFRESH, INTENSITY, INDEPENDENT, NONE }
enum InputMode { GAMEPLAY, UI }
enum InputDevice { KEYBOARD_MOUSE, GAMEPAD }
enum RoomType { COMBAT, ELITE, REST, REWARD, SHOP, BOSS, ENTRY }
enum RunOutcome { DEATH, VICTORY }
```

## Alternatives Considered

### Alternative 1: JSON/YAML Data Files
- **Description**: Store game data in JSON or YAML, parse at runtime
- **Pros**: Language-agnostic, easy to diff in git, scriptable generation
- **Cons**: No editor integration (can't drag-drop, no inspector), no type safety, manual serialization code, no hot-reload
- **Rejection Reason**: Godot Resources ARE the data pipeline. JSON loses every Godot-specific benefit.

### Alternative 2: GDScript Constants
- **Description**: Define cards as dictionaries in GDScript files
- **Pros**: Type-checked at parse time, no file I/O
- **Cons**: Not editor-authorable, designers must edit code, no visual inspector, scales poorly to 100+ cards
- **Rejection Reason**: Violates "data-driven" requirement. Card authors should never touch GDScript.

## Consequences

### Positive
- Full Godot editor integration — designers can author cards, enemies, and effects in the inspector
- Hot-reload: modifying a .tres file updates the game immediately during play-testing
- Type safety: exported Resource fields have editor validation
- `duplicate_deep()` handles the one tricky case (nested resource copying)

### Negative
- .tres files are Godot-specific — no external tooling reads them without Godot
- Large numbers of .tres files (100+ cards) may need editor organization tooling

### Risks
- **Accidental shared references**: If two systems hold the same CardData reference and one modifies it, the other sees the change. Mitigation: CardData is read-only at runtime (no system writes to loaded Resources). Deep copy only when runtime modification is explicitly needed.

## Performance Implications
- **CPU**: Resource loading at startup (CardRegistry loads all cards): <100ms for 25 .tres files. Negligible.
- **Memory**: 25 CardData resources × ~2KB each = ~50KB. Status effects and enemies add ~20KB. Total data < 1MB.
- **Load Time**: All resources loaded once at CardRegistry._ready(). No per-room data loading.

## Migration Plan
No existing code — greenfield implementation.

## Validation Criteria
1. A CardData .tres file created in the Godot editor loads correctly via `CardRegistry.get_card()`
2. `CardData.duplicate_deep()` produces a copy where modifying a nested CardEffect does not affect the original
3. All enum values used in .tres files match the Enums autoload definitions
4. Adding a new card requires zero code changes — only a new .tres file in the correct directory
